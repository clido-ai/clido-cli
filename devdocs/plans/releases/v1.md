# Clido V1 Plan

## Goal

Ship the first version of Clido that is genuinely useful on real repositories.

V1 should already be able to:
- accept a task from the CLI
- inspect files and repositories
- call tools in a loop
- modify files safely enough for normal development work
- resume work after interruption

## Product Definition

V1 is not a prototype. It is the first release that should feel like a real coding agent for one developer on one machine.

The focus is not breadth. The focus is a strong core loop.

## Included Scope

### Foundation

From the main roadmap:
- Phase 1 complete
- Phase 2 complete
- Phase 3 complete

This gives Clido:
- workspace structure and shared types
- one working provider path
- the agent loop
- the six core tools: `Read`, `Write`, `Edit`, `Glob`, `Grep`, `Bash`
- a proper CLI (including `--profile`, `--provider`, `--model` flags)
- session storage
- config loading with **named profiles**, **project-level config** (`.clido/config.toml`), and **clear API key error messages**
- streaming output

### Required quality features

Also include these milestone groups before calling V1 done:
- Phase 4.2 — Context Engine
- Phase 4.3 — Permission System
- Phase 4.5 — Plan Mode
- Phase 5.1 — Robust Error Handling
- Phase 5.2 — Session Recovery
- Phase 5.3 — Graceful Shutdown
- Phase 5.4 — Integration Test Suite
- Phase 8.4 (basic) — `clido doctor` (API key, session dir, pricing.toml presence and staleness)

## Why These Are In V1

Without these additions, Clido would still be too fragile:
- no context compaction means long tasks degrade quickly
- no permission model makes the agent unsafe by default
- no session recovery makes interruptions painful
- no integration suite makes regressions too easy

## Explicitly Out of Scope

Do not put these into V1:
- multi-provider support
- subagents
- planner / task graph
- memory system
- repository indexing
- MCP support
- sandboxing
- audit logging
- live plan visualization

Those features may be valuable, but they are not required to prove that Clido is a strong agent.

## Release boundaries (canonical)

- **Named profiles and `--profile` / `--model` / `--provider` flags are V1.** They are config/CLI surface; only the Anthropic provider is implemented in V1. Specifying a non-Anthropic profile (e.g. `openrouter`) in V1 returns a helpful error at startup, not a runtime panic.
- **`clido doctor` is V1** with basic checks (API key, session dir, pricing.toml presence and staleness). It is expanded in V1.5 (e.g. MCP checks, provider connectivity ping).
- **`pricing.toml` and its staleness warning are V1.** Cost *tracking* (cumulative session cost) is V1.5, but the pricing file and the 90-day staleness check ship in V1 because doctor depends on them.

## CLI Surface

The canonical CLI for every release is defined in [cli-interface-specification.md](../cli-interface-specification.md). This section summarizes the V1 surface.

**Commands and subcommands (V1):**
- `clido <prompt>` — run with positional prompt
- `clido` (no args, TTY) — interactive REPL
- `clido --version` / `clido version`
- `clido --continue`, `--resume <id>`, `--print` / `-p`
- `clido --tools`, `--allowed-tools`, `--disallowed-tools`, `--permission-mode`
- `clido --profile`, `--model`, `--provider`
- `clido --system-prompt`, `--system-prompt-file`, `--append-system-prompt`
- `clido --max-turns`, `--max-budget-usd`, `--output-format`, `--no-color`, `--verbose`
- `clido sessions list`, `clido sessions show <id>` (legacy: `list-sessions`, `show-session` with deprecation notice)
- `clido init`

**UX outcomes (V1):**
- Rich streaming text output with tool lifecycle (pending → in progress → completed)
- Session list/show with ID, date, turns, cost, preview
- Permission prompts for state-changing tools; plan mode and ExitPlanMode tool
- First-run auto-detection when no config exists (interactive); `clido init` for explicit setup
- Exit codes: 0 success, 1 runtime error, 2 usage/config, 3 soft limit (budget/turns)
- **UX spec:** First-run and init follow [ux-requirements.md](../ux-requirements.md): prompts are specified copy, state what to type and press Enter; CLI is functional and visually clear ("hübsch"); scripts that run interactive commands print an intro (e.g. run-in-test-env.sh before `clido init`).

## Exit Criteria

V1 is done when all of the following are true:
- `clido` can complete multi-step repository tasks using the six core tools
- long sessions can compact context without collapsing quality; compaction failure falls back gracefully rather than crashing
- users can run in normal mode or plan-only mode
- state-changing tool calls are permission-gated; concurrent tool calls that trigger `AskUser` are serialized (one prompt at a time, no deadlock)
- interrupted sessions can be resumed; stale-file detection warns on files changed externally since last session
- core end-to-end scenarios pass reliably
- **Coverage floor met:** workspace-wide line coverage ≥ 70 % (measured by `cargo-tarpaulin`); see Phase 5.4.2 for per-crate targets
- **V1 critical-path tests all pass** (agent loop, provider, permissions, session, config — see Phase 5.4.2)
- Configuring a profile for an unimplemented provider (e.g. `openrouter` in V1 binary) produces a helpful error, not a runtime panic
- `clido doctor` passes all V1 checks (API key, session dir, `pricing.toml`; connectivity ping is V1.5)
- `pricing.toml` is present and reports a warning if the file is older than 90 days

## Suggested Internal Milestones

1. Build the workspace, types, provider, and core loop.
2. Finish all six core tools.
3. Add proper CLI, config, sessions, and streaming.
4. Add context builder plus compaction.
5. Add permission checks and plan mode.
6. Harden error handling and recovery.
7. Lock in integration coverage.

## Success Standard

If V1 is successful, Clido should already be credible for tasks like:
- auditing a repository
- locating code paths and editing them
- fixing simple build or test issues
- creating or updating project files
- resuming an interrupted task without losing context
