# Clido V2 Plan

## Goal

Turn Clido from a strong single-provider local tool into a complete product that other people can adopt and trust.

V2 is where Clido becomes easier to install, safer to run, less tied to one provider, and measurable in its behavior.

## Included Scope

From the main roadmap:
- Phase 4.1 — Multi-Provider Support
- Phase 4.2.4 — Prompt Caching (Anthropic)
- Phase 6.1 — Startup Performance
- Phase 6.3 — Concurrent Provider Requests
- Phase 6.4 — File Read LRU Cache
- Phase 7.1 — Bash Sandboxing
- Phase 7.3 — Audit Logging
- Phase 8.1 — Hooks System
- Phase 8.5 — Shell Completion and Man Pages
- Phase 8.6 — Live Plan / Progress Visualization
- Phase 9.1 — Full Test Coverage
- Phase 9.2 — Benchmarks
- Phase 9.3 — Structured Telemetry
- Phase 9.4 — Documentation
- Phase 9.5 — Packaging and Distribution
- Phase 9.6 — Production Hardening

## Why This Is V2 And Not Earlier

These features are important, but each of them requires a proven core to build on.

Multi-provider support requires a working agent loop and a tested provider abstraction. Prompt caching requires a working multi-turn loop (to verify cache hits) and the complete system prompt architecture — it would be premature in V1 when the context engine is still being defined. Sandboxing requires a working Bash tool and a tested permission model. Telemetry requires a stable event model. Documentation requires stable behavior to describe. Shell completion requires a stable CLI surface.

None of these should be built before the agent itself is stable. V2 is the productization phase.

## Why These Are Not In V3 Or Later

Phase 6.3 (Concurrent Provider Requests) is a performance feature, not a research feature. It belongs with performance work in V2.

Phase 8.5 (Shell Completion) and 8.6 (Live Plan Visualization) are developer-experience features that users expect from a mature CLI tool. They do not require advanced agent capabilities.

Phase 9.3 (Telemetry) belongs with production hardening, not as an afterthought.

## Expected User Value

By the end of V2, Clido should be:
- installable via standard package managers on macOS and Linux
- usable with at least Anthropic, OpenAI, and OpenRouter providers — switchable via `--profile`, `--provider`, and `--model` flags without editing config
- **60–80% cheaper on input tokens** for multi-turn Anthropic sessions via prompt caching
- measurably faster at startup and repeated reads than V1
- safer to run on production codebases with Bash sandboxing
- observable through structured telemetry and an audit log
- debuggable via shell completion and visible progress output
- documented well enough that a new user can self-serve

## Explicitly Out of Scope

Still deferred:
- subagents
- planner / task graph
- memory system
- MCP support
- repository indexing

## CLI Surface

The canonical CLI is defined in [cli-interface-specification.md](../cli-interface-specification.md). V2 adds the following to the V1.5 surface.

**New in V2:**
- `clido run <prompt>` — explicit run subcommand
- `clido --input-format text|stream-json` — for SDK/subprocess integration
- `clido --sandbox` — Bash sandboxing
- `clido audit`, `clido audit --tail`, `--session`, `--tool`, `--since`, `--json`
- `clido stats`, `clido stats --session <id>`, `--json`
- `clido completions bash|zsh|fish`, `clido man`

**UX outcomes (V2):**
- Provider switching via `--profile` / `--provider` / `--model` without editing config
- Shell completion and man page
- Audit log and stats commands with tabular and `--json` output
- Live plan / progress visualization in interactive mode
- All interactive prompts and scripts follow [ux-requirements.md](../ux-requirements.md); CLI remains functional and visually clear ("hübsch").

## Exit Criteria

V2 is done when all of the following are true:
- Anthropic, OpenAI, and OpenRouter providers all work behind the same `ModelProvider` trait
- prompt caching is active on Anthropic: a 20-turn session shows `cache_read_input_tokens > 0` and the session cost reflects the 10% rate for cached tokens
- Bash execution can run in sandboxed mode on both macOS and Linux
- startup completes in under 200ms to first API request
- file read LRU cache reduces repeated read latency measurably
- concurrent provider requests do not regress sequential correctness
- every tool call is recorded in an append-only audit log
- structured telemetry records tool durations, provider calls, and retry events
- hooks fire for PreToolUse and PostToolUse when configured
- shell completion works in bash and zsh
- live plan or progress output is visible in interactive mode
- test coverage meets the targets defined in `devdocs/guides/testing-strategy-and-master-test-plan.md`
- benchmarks are tracked and a regression baseline exists
- documentation covers installation, configuration, and all CLI flags
- a release binary can be installed on a clean macOS and Linux system

## Suggested Internal Milestones

1. Implement OpenAI and OpenRouter providers; build provider factory.
2. Implement prompt caching on Anthropic provider; verify cost reduction in multi-turn tests.
3. Implement concurrent provider request support with correctness tests.
4. Implement file read LRU cache; profile startup time.
5. Implement Bash sandboxing (macOS sandbox-exec, Linux seccomp / Docker).
6. Implement audit logging and structured telemetry.
7. Implement hooks system (PreToolUse, PostToolUse).
8. Add shell completion and man pages.
9. Implement live plan / progress visualization.
10. Reach test coverage targets; build benchmark baselines.
11. Write documentation; build release packages; validate on clean systems.

## Release Intent

This is the first version suitable for broader internal or external adoption. After V2, Clido should feel like a finished product, not a well-built tool.
