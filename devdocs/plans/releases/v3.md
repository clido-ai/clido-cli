# Clido V3 Plan

## Goal

Add advanced capability layers that materially improve difficult tasks on top of a stable, proven base.

V3 is where Clido moves from a strong CLI tool to a serious agent platform.

## Included Scope

From the main roadmap:
- Phase 4.7 — Subagent Architecture
- Phase 4.9 — Workflow Engine (Declarative Execution Layer)
- Phase 4.9.x — Pre-Flight und dynamische Parameter
- Phase 5.5 — Memory System
- Phase 8.3 — MCP Support
- Phase 8.7 — Repository Indexing

User-defined YAML workflows enable repeatable pipelines (e.g. code review, release checklists) and ship before or alongside the V4 planner.

## Why These Features Are Deliberately Delayed

Each system here is genuinely useful, but each one also introduces significant complexity that can degrade the baseline if it is built before the foundation is stable.

- **Subagents** create concurrency, orchestration, and session isolation challenges. They require the V1.5 semaphore, the V2 audit log, and the V1 permission model to be solid before subagent execution can be safely bounded.
- **Memory** can improve continuity across sessions, but can also pollute context if retrieval is noisy or relevance is weak. Building memory before compaction is proven creates compounding risks. The V3 memory system uses a hybrid retrieval strategy (FTS5 keyword search + `sqlite-vec` vector similarity via `fastembed-rs` local embeddings) with relevance-based context injection — not recency. This requires that context compaction (V1) and the context engine (V1) are stable and tested before layering memory on top.
- **MCP support** expands tool reach significantly, but increases integration surface area proportionally. It should be added after the core tool model is stable and well-tested.
- **Repository indexing** only justifies its overhead when search quality is strong enough to outperform Grep and Glob on real repositories. That quality threshold is hard to reach cheaply.

## The V3 Design Rule

Do not ship these systems because they are impressive.

Ship them only if they measurably improve real task success rates compared to Clido without them.

For each V3 feature, define before implementation:
- what specific class of tasks it is intended to improve
- what metric will be used to measure success
- what the fallback behavior is if the feature underperforms or degrades reliability

If the measurement criteria cannot be defined in advance, the feature is not ready for V3.

## Note On Subagent Concurrency

Subagents require concurrent agent execution. The bounded concurrency semaphore from Phase 4.6 (V1.5) handles tool-level parallelism. Subagent-level concurrency is a separate concern addressed in Phase 4.7. Both must work correctly before parallel subagents can run safely.

## CLI Surface

The canonical CLI is defined in [cli-interface-specification.md](../cli-interface-specification.md). V3 adds the following to the V2 surface.

**New in V3:**
- `clido memory list`, `clido memory prune`, `clido memory reset`
- `clido fetch-models`
- `clido workflow run`, `workflow check`, `workflow validate`, `workflow inspect`, `workflow list`
- **SemanticSearch** tool — available to the model (query, optional target_directory, num_results); uses embedding engine and repo indexing

**UX outcomes (V3):**
- Memory commands for listing, pruning, and resetting cross-session memory
- Workflows loadable from YAML; pre-flight checks step-tools, profiles, API keys, and optional prerequisites; `workflow check` runs before first run
- Semantic search tool available in the agent loop (no SemanticSearch in V1 or V2)
- All interactive prompts and scripts follow [ux-requirements.md](../ux-requirements.md); CLI remains functional and visually clear ("hübsch").

## Exit Criteria

V3 is done when all of the following are true:
- subagents can run isolated reasoning tasks without destabilizing or corrupting the parent session
- workflows can be loaded from YAML and executed; pre-flight validates step-tools, profiles, API keys, and optional prerequisites; `workflow check` can be run before the first execution
- memory retrieval produces relevant results more often than it introduces noise: hybrid search (FTS5 + vector) is validated against a fixture task suite; semantic retrieval finds paraphrased memories that FTS5 alone would miss; irrelevant memories for an unrelated task do NOT surface in context
- MCP servers can be configured, started, and safely used as registered tools
- repository indexing measurably improves search or navigation on medium-to-large repositories compared to Grep and Glob alone

## Suggested Internal Milestones

1. Define measurement suite for each V3 feature before implementation begins.
2. Build SubAgentManager and session isolation; validate parent session correctness.
3. Implement `EmbeddingEngine` (fastembed-rs, local ONNX); validate embedding quality with similarity tests.
4. Implement long-term memory store: SQLite + FTS5 + sqlite-vec hybrid; validate keyword, semantic, and hybrid retrieval.
5. Wire relevance-based memory injection into agent loop; validate that injected memories match task context.
6. Build MCP client and server lifecycle management; validate against at least two real MCP servers.
7. Build file and symbol index; validate search quality against fixture repositories.
8. Run full V3 measurement suite; confirm each feature meets its success threshold.

## Release Intent

V3 is for advanced users who want more than a strong baseline agent and are willing to accept more system complexity in exchange for real capability gains on hard tasks.
