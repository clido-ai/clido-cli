# Clido V4 Plan

## Goal

Reserve the highest-complexity and most experimental capabilities for a stage where Clido is already mature, stable, and measurable.

V4 is intentionally narrow in scope but high in ambition.

## Included Scope

From the main roadmap:
- Phase 4.8 — Task Graph / Planner
- any future experimental orchestration layer that depends on a proven V1 through V3 foundation

Note: Phase 6.3 (Concurrent Provider Requests) was moved to V2. It is a performance feature, not an experimental one, and belongs with the rest of the performance work.

## Why These Features Are Last

The task graph planner is an optional overlay on top of the reactive agent loop. It does not replace the loop. It pre-structures work before the loop runs.

That makes it valuable for a narrow but important class of tasks:
- large, well-defined refactors with many steps
- parallel analysis across many files that must be combined
- complex workflows where upfront planning reduces wasted turns

It is not valuable for open-ended exploration or short tasks. For those, the reactive loop is better.

This means the planner must be proven to improve specific task types before it is worth the cost and complexity. That proof requires a mature, measurable baseline — which is why V4 comes last.

## Preconditions For V4

V4 should not start until:
- the reactive loop has been running reliably for long enough to have real quality data
- benchmark baselines from V2 and measurement suites from V3 are established
- there is a specific set of tasks where the reactive loop demonstrably underperforms and planning would help

If those preconditions are not met, V4 should wait.

## CLI Surface

The canonical CLI is defined in [cli-interface-specification.md](../cli-interface-specification.md). V4 adds the following to the V3 surface.

**New in V4:**
- `clido --planner` — enable task graph / planner mode for the run

**UX outcomes (V4):**
- Planner mode optional; fallback to reactive loop when planner is not used or graph is invalid
- Live plan display when planner is active (per spec §5 and development-plan Phase 8.6)
- All interactive prompts and scripts follow [ux-requirements.md](../ux-requirements.md); CLI remains functional and visually clear ("hübsch").

## Guardrails

Before shipping V4 features:
- define a benchmark suite comparing reactive-only versus planner-guided execution on the same tasks
- require that planner-guided execution wins on task success rate, not just on a subjective impression
- require graceful fallback to the reactive loop when the planner produces an invalid or low-quality graph
- confirm that adding the planner does not regress reliability for any V1–V3 task class

## Exit Criteria

V4 is done when all of the following are true:
- planner-based execution improves task success rate on the identified task class by a measurable margin
- fallback to the reactive loop is seamless and tested
- the planner is strictly optional — no existing workflow depends on it
- adding `--planner` to any V1–V3 task does not regress its outcome

## Suggested Internal Milestones

1. Define the target task class and success benchmark for the planner.
2. Implement TaskGraph types and DAG validation.
3. Implement planner model call and graph parsing.
4. Implement deterministic DAG executor with dependency resolution.
5. Implement fallback: invalid graph → reactive loop.
6. Run benchmark comparison and confirm the planner wins on the target task class.
7. Gate behind `--planner` flag; confirm no impact on default behavior.

## Release Intent

V4 is for high-end workflow optimization on specific, identified task types.

Clido should already be an excellent agent before anything in this document ships. The planner is not what makes Clido good. It is what makes Clido better at a specific class of hard problems after everything else is already proven.
